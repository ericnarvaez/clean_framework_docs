• Multiple entities refers to when a feature or use case involves more than just one primary entity

• The question that you should be asking yourself is.... "How should I be structuring this?"

• The Savings Account feature is an example of a feature with many entities

• The guidance that is being given is to structure the entities under a primary "Root" entity

• All entities objects should be associated to a "Root" hierarchy

      - SavingAccountRootEntity        (Primary "Root" Entity)
       - List<Payee>            (Sub Entity)
       - List<SavingAccount>       (Sub Sister Entity)
        - SavingAccount         (...Nested Entity)
           - AutomaticSavingsInfo     (...Nested Entity)
           - FullAccountNumber      (...Nested Entity)
           - List<PendingTransaction>   (...Nested Entity)
            - PendingTransaction     (...Nested Entity)
           - List<HistoryTransaction>    (...Nested Entity)
            - HistoryTransaction     (...Nested Entity)
           - HistoryTransaction      (...Nested Entity)
           - CheckImage         (...Nested Entity)
        - Search             (...Nested Sister Entity)
           - Payee            (...Nested Entity)
           - List<HistoryTransaction>    (...Nested Entity)
    • It is up to ServiceAdapters to update a "Root" entity's nodes(nested entities)
    • It's also expected that the Entity Model (one of the Typed Parameter) for all of the ServiceAdapters used in the usecase would be a Root entity as well
